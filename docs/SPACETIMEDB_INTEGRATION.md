# SpacetimeDB Integration Research

## Snapshot

SpacetimeDB is a real-time backend/database for apps and games. It combines database tables, server-side module logic, reducer calls, real-time row subscriptions, persistence, and generated type-safe client bindings.

Current public signals checked on 2026-04-24:

- Main repo: `clockworklabs/SpacetimeDB`
- Latest GitHub release observed: `v2.1.0`
- Current npm package for client/module bindings: `spacetimedb@2.1.0`
- Deprecated npm package: `@clockworklabs/spacetimedb-sdk` now says to use `spacetimedb`
- Browser/Vite usage is supported by the npm package
- Server modules can be written in TypeScript, C#, C++, or Rust

## Core model

SpacetimeDB concepts map cleanly to a multiplayer game backend:

- **Host**: the SpacetimeDB server or managed maincloud instance
- **Database**: the running application/database on a host
- **Module**: server-side schema plus business logic
- **Tables**: persistent in-memory data, automatically persisted to disk
- **Reducers**: transaction-wrapped functions clients call to mutate state
- **Subscriptions**: client-side real-time replicas of selected rows
- **Generated bindings**: type-safe client SDK generated from the module schema

The important detail for Ashen-Hallow: clients should not write rows directly. They call reducers. Reducers validate commands and mutate tables atomically.

## Fit for Ashen-Hallow

SpacetimeDB is a strong fit for:

1. multiplayer lobby and presence
2. saved match history
3. cloud persistence for player profiles and runs
4. server-authoritative PvP commands
5. real-time synced match state
6. eventually, ranked or asynchronous battles

It should **not** replace the current local deterministic engine immediately. The current pure TypeScript architecture is valuable and should stay as the gameplay core while SpacetimeDB is introduced as an adapter/backend layer.

## Recommended integration strategy

### Phase 1: Lobby and presence proof of concept

Goal: prove connection, auth identity, generated bindings, subscriptions, and reducers without touching battle determinism.

Add:

```text
spacetimedb/
  package.json
  src/index.ts

src/infra/spacetimedb/
  SpacetimeConnection.ts
  SpacetimeConfig.ts
  LobbySyncService.ts
  module_bindings/        # generated by `spacetime generate`
```

SpacetimeDB tables:

```text
player_session
- identity
- display_name
- connected_at_ms
- last_seen_at_ms

lobby
- id
- status: OPEN | IN_MATCH | CLOSED
- created_by_identity
- created_at_ms

lobby_member
- lobby_id
- identity
- race: HUMAN | ORC | null
- ready: boolean
```

Reducers:

```text
upsert_player_session(display_name)
create_lobby()
join_lobby(lobby_id)
leave_lobby(lobby_id)
set_lobby_race(lobby_id, race)
set_lobby_ready(lobby_id, ready)
```

Client integration:

- add `spacetimedb` dependency
- connect from browser using generated `DbConnection`
- subscribe to lobby and lobby_member tables
- show lobby UI before `GameScreen`
- keep current single-player path untouched

Why first: low risk, immediately useful, and validates the whole SpacetimeDB toolchain.

### Phase 2: Match persistence and action logs

Goal: persist matches without making the server authoritative yet.

Tables:

```text
match_record
- id
- created_at_ms
- player_identity
- player_race
- enemy_race
- status
- winner
- ended_at_ms

action_log
- match_id
- sequence
- turn
- action_type
- payload_json
- created_at_ms

round_summary
- match_id
- round
- winner
- player_damage
- enemy_damage
- payload_json
```

Reducers:

```text
create_match_record(player_race)
append_match_action(match_id, sequence, action_type, payload_json)
commit_round_summary(match_id, round, payload_json)
finish_match(match_id, winner)
```

Current project integration points:

- wrap `Store.dispatch` with an optional action-log sink
- serialize safe `GameAction` payloads from `src/engine/game/actions.ts`
- persist `lastRoundSummary` from `GameState`
- do not send high-frequency `TICK` actions at first

This gives cloud saves/replays while keeping local gameplay reliable.

### Phase 3: Remote command mode

Goal: allow two clients to submit commands to the same match.

Model:

- browser sends user intent reducers like `place_unit`, `unlock_unit`, `ready`
- SpacetimeDB validates and stores accepted commands
- clients subscribe to command/match-state rows
- local game applies accepted commands in deterministic order

Tables:

```text
match_command
- match_id
- sequence
- identity
- command_type
- payload_json
- accepted_at_ms
```

Reducers:

```text
submit_match_command(match_id, command_type, payload_json)
acknowledge_client_state(match_id, local_checksum)
```

Important guardrail:

- command sequence must be server assigned
- clients must not decide order independently
- add checksums to detect desyncs

### Phase 4: Server-authoritative battle simulation

Goal: SpacetimeDB becomes the real match authority.

This is the biggest step. There are two viable routes:

#### Option A: TypeScript SpacetimeDB module

Pros:

- same language as current engine
- easiest conceptual migration from existing TypeScript types
- faster for Ael's current codebase

Cons:

- current Vite/browser code cannot be imported wholesale into a server module
- the core engine must stay pure, DOM-free, and package-friendly
- generated bindings and module code need strict boundaries

Needed refactor:

```text
packages/game-core/       # pure deterministic game logic
spacetimedb/              # SpacetimeDB module imports game-core where possible
src/                      # Vite client imports game-core and generated bindings
```

#### Option B: Rust module

Pros:

- strongest fit for deterministic, high-performance server simulation
- likely best long-term for server-authoritative PvP

Cons:

- requires porting game rules from TypeScript to Rust
- higher upfront cost
- more duplicate model risk unless data definitions are generated/shared

Recommendation: start with **TypeScript module POC**, then only move to Rust if performance or deployment constraints demand it.

## Client architecture changes

Add an adapter layer. Do not wire SpacetimeDB directly into `GameScreen`.

```text
src/infra/spacetimedb/
  SpacetimeConnection.ts       # DbConnection setup, token storage, connect/disconnect
  LobbySyncService.ts          # lobby subscriptions and reducers
  MatchSyncService.ts          # match subscriptions and command reducers
  SpacetimeActionMapper.ts     # GameAction <-> backend command payloads

src/engine/sync/
  LocalGameSession.ts          # current behavior
  RemoteGameSession.ts         # remote command flow
  GameSession.ts               # shared interface used by UI
```

Target interface:

```ts
export interface GameSession {
  getState(): GameState;
  subscribe(listener: (state: GameState) => void): () => void;
  dispatch(action: GameAction): void;
  destroy(): void;
}
```

Then `GameScreen` receives a `GameSession` instead of constructing `Store` directly. That lets the UI work in local or remote mode.

## Tooling commands

Install CLI:

```bash
curl -sSf https://install.spacetimedb.com | sh
```

Install client package:

```bash
npm add spacetimedb
```

Create a TypeScript module POC:

```bash
spacetime dev --template basic-ts
```

Generate TypeScript bindings:

```bash
mkdir -p src/infra/spacetimedb/module_bindings
spacetime generate --lang typescript \
  --out-dir src/infra/spacetimedb/module_bindings \
  --module-path spacetimedb
```

Connect from Vite client:

```ts
import { DbConnection, tables } from './module_bindings';

export const connectSpacetime = (token?: string): DbConnection =>
  DbConnection.builder()
    .withUri(import.meta.env.VITE_SPACETIMEDB_URI ?? 'ws://localhost:3000')
    .withDatabaseName(import.meta.env.VITE_SPACETIMEDB_DB ?? 'ashen_hallow_dev')
    .withToken(token ?? '')
    .onConnect((ctx, identity, nextToken) => {
      localStorage.setItem('ashen-hallow:spacetime-token', nextToken);
      ctx.subscriptionBuilder().subscribe([tables.lobby, tables.lobbyMember]);
      console.info('SpacetimeDB connected', identity.toHexString());
    })
    .onConnectError((_ctx, error) => {
      console.error('SpacetimeDB connection failed', error);
    })
    .build();
```

## Testing plan

Add tests before implementation:

1. action serialization tests
   - `GameAction` converts to backend command payload
   - invalid/unsupported actions are rejected

2. remote session adapter tests
   - local dispatch path remains unchanged
   - remote mode queues/sends command instead of mutating locally

3. reducer parity tests
   - server reducer accepts the same placement constraints as local reducer
   - command ordering is deterministic

4. integration smoke test
   - connect local client to local SpacetimeDB
   - create lobby
   - join lobby
   - set race
   - observe table updates through subscription

5. CI guardrail
   - generated bindings exist
   - SpacetimeDB integration stays behind `src/infra/spacetimedb`
   - current `npm test` remains green without requiring a live SpacetimeDB server

## Risks and mitigations

| Risk | Mitigation |
|---|---|
| Server/client game logic diverges | Keep local core pure, add parity tests, move shared rules to `packages/game-core` before authoritative mode |
| Too much network traffic during battles | Sync commands and coarse snapshots first, not every animation frame |
| GitHub Pages cannot host backend | Use SpacetimeDB maincloud or a self-hosted SpacetimeDB server, client remains static on Pages |
| Generated bindings churn | Commit generated bindings only after module schema stabilizes, add docs for regeneration |
| Auth confusion | Start with SpacetimeDB identity token, add external auth later only if needed |
| Reducer validation gaps | Treat SpacetimeDB reducers as the only trusted write path and duplicate placement/economy validation server-side |

## Recommendation

Do this in order:

1. Add a SpacetimeDB lobby/presence POC.
2. Add action-log and match-summary persistence.
3. Add remote command mode with server-assigned sequencing.
4. Only then decide if the full battle sim moves into a TypeScript SpacetimeDB module or gets ported to Rust.

For the current Ashen-Hallow codebase, the best first implementation is **SpacetimeDB as a backend adapter, not a full engine replacement**.
